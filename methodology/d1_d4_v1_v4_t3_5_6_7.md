## D2 — Structural Analysis (delta)

**New resource axis**: TMEM accumulator. On B200 (sm_100a), each SM has 256 KB TMEM.
- QK^T accumulator: `M × N × 4B` = `64 × 64 × 4` = 16 KB
- PV accumulator: `M × HEAD_DIM_CKV × 4B` = `64 × 512 × 4` = **128 KB**
- Combined: 144 KB per CTA → occupancy capped at 1 CTA/SM (tighter than existing
  `__launch_bounds__(512,1)`).

**New SMEM requirement** for MMA operand layout (8×T core-tile per GT-11 layout,
re-derived for kind::f16 K=16):
- Q SMEM tile: `M × K_total × 2B` = `64 × 576 × 2B` = 72 KB (needs to be resident
  for the full MMA K-loop; 36 K-tile issues × K=16)
- K SMEM tile per prefetch buffer: `N × K_total × 2B` = `64 × 576 × 2B` = 72 KB
  per buffer. With 2-buffer prefetch that's 144 KB.
- Total SMEM: 72 + 144 = **216 KB per CTA**.

B200 SMEM limit is ~228 KB per SM. So SMEM fits 1 CTA/SM, same occupancy bound.

**V2**: consistent. The TMEM + SMEM budget is feasible but tight; further
softmax scratch on-SMEM becomes infeasible.

## D3 — Hardware Binding (core regime re-analysis)

### D3.1 — Arithmetic Intensity re-derivation

Per-CTA, N=8 large workload:
- **Bytes gathered from HBM** per CTA: 128 KV rows × 1152 B/row = **147 KB**.
- **FLOPs per CTA** (ignoring softmax overhead):
  - QK^T: `M_real × K × N_total` × 2 = 16 × 576 × 128 × 2 = **2.36 MFLOP**
  - attn@Kc: 16 × 128 × 512 × 2 = **2.10 MFLOP**
  - Softmax: negligible (`~16 × 128` exp + adds = 0.005 MFLOP)
  - Total: ~4.5 MFLOP per CTA.

Arithmetic intensity = 4.5e6 / 147e3 ≈ **30 FLOP/byte**.

**B200 ridge points:**
- Scalar fp32: ~80 TFLOP/s / 7 TB/s ≈ **11 FLOP/B** ← A23 operates here
- BF16 tensor-core: ~5 PFLOP/s / 7 TB/s ≈ **714 FLOP/B**

At AI=30, kind::f16 MMA puts the kernel **firmly memory-bound by 23×**. Moving
to tensor cores does not increase the achievable throughput because HBM
is the ceiling. Any compute speedup is below the memory noise floor.

### D3.2 — Padding waste

`kind::f16` cta_group::1 dense MMA requires M ∈ {64, 128}. Our M_real = 16.
Three options:

**Option A — pack 4 tokens per CTA (M=64, 4 real tokens)**:
- Destroys split-K grid structure. S scales as `ceil(N/4)`:
  - N=1 → S=1 CTAs, or N=1 with 3 zero-pad tokens → launch (1, 1) = 1 CTA (1/132 SMs)
  - N=8 → 2 "macro-tokens", S ≤ 2 → ~2-4 CTAs (3% SM utilization)
- Directly contradicts GT-30 ("sparse-attention MLA decode is fundamentally split-K").
- Rejected: catastrophic SM under-utilization for N ≤ 8.

**Option B — FP8 KV via kind::f8f6f4**:
- Spec mandates bf16 KV (CLAUDE.md §1). Dequantization on the fly keeps bf16 loads
  → same bandwidth, same regime.
- If we stored KV as FP8 in a *modified* cache, that violates the harness contract.
- Rejected.

**Option C — Pad M=16 → 64 via zero rows within a single CTA**:
- Real rows: 16 heads. Padded rows: 48 zeros.
- **Compute waste: 4×** on MMA (75% of FLOP useless).
- **TMEM waste: 4×** (accumulator for 64 rows, only 16 kept).
- **Memory not wasted** (Q loaded once as 16 rows actual, replicated or zero-padded
  in SMEM layout, but the KV tile is shared across M-rows — no extra fetches).
- **SMEM layout reorganization required**: current per-lane interleaved layout (A18)
  matches scalar dot product; MMA requires 8×T core-tile format per GT-11 with
  K-major SMEM. Q, K, and post-MMA output all need new layouts. Lane-to-TMEM mapping
  per GT-12 differs from current interleaved per-lane register mapping.

### D3.3 — Implementation cost of Option C

End-to-end per-CTA additions (minimum viable tcgen05 kernel):

| Component | Additional code | Additional complexity |
|---|---|---|
| `tcgen05.alloc` / dealloc lifecycle | ~40 lines | TMEM buffer address tracking, dealloc-on-all-exits invariant |
| TMA 2D tensor-map setup for Q + K (need tensormap.replace per GT-6) | ~120 lines | Residency rules (tensormap in global, not shared), modification flow, cp_fenceproxy |
| SMEM Q layout (8×T core-tile, K-major) | ~60 lines | Fill formula from GT-11, re-derive SBO/LBO for `kind::f16` M=64 N=64 K=16 via **new sbo_lbo_sweep probe** (not confirmed for kind::f16; GT-11 only covers kind::f8f6f4) |
| SMEM K layout (same core-tile format per prefetch buffer) | ~40 lines | K dim is 576, needs K-tile splitting |
| IDESC for QK^T MMA (kind::f16, M=64 N=64 K=16) | ~30 lines | New transpose_A/transpose_B bits; re-derive per GT-15 — not known for kind::f16 |
| QK^T MMA issue loop (36 K-tile issues) | ~50 lines | elect_one_sync, commit+mbar wait per GT-2, fence::after_thread_sync |
| TMEM→register load of QK^T logits (64×64 tile) | ~50 lines | GT-17 full-warp participation; lane distribution for M=64 kind::f16 (not previously confirmed) |
| Softmax on register logits + mask padded M rows | ~80 lines | Per-row max, exp, sum; drop rows 16..63 |
| Re-pack P (softmax output) for PV MMA | ~60 lines | Either TMEM→SMEM (bf16) for next MMA, OR use TMEM-A-operand MMA variant |
| PV MMA issue loop (K=64 N=N_current contracted to HEAD_DIM_CKV=512) | ~60 lines | New IDESC for PV — different operand layouts from QK^T |
| TMEM→register load of PV output + masked store | ~60 lines | 64×512 output tile; per-warp partition; mask padded M rows |
| Warp specialization split (producer vs consumer) | ~100 lines | mbarrier dual-role (producer arrive + consumer wait); warp_id dispatch |
| Integration with current split-K + last-arrival reduce | ~50 lines | TMEM must be dealloc'd before atomicAdd; counter reset path unchanged |

**Total**: ~800-1000 lines of new code. Prior sbo_lbo_sweep probe on the top-k
project took **5 Modal runs** to converge on the correct SBO/LBO values for
`kind::f8f6f4` M=64 N=64 K=32. `kind::f16` K=16 has a different layout stride
so the sweep must be re-run from scratch. Lane distribution for M=64 kind::f16
similarly untested (GT-12 was confirmed for kind::f8f6f4 only).

### D3.4 — Expected wall-time outcome

Per-CTA main-loop breakdown at A23 (scalar fp32 path):
- 32 outer iters × ~55 ns/iter active = **~1.8 µs** per CTA main loop
- Per-iter active work: ~80% scalar compute, ~20% SMEM load + mbarrier (TMA hidden).

Per-CTA main-loop at Option C (tcgen05 MMA path):
- 2 outer iters (N=64 MMA tile) × (QK^T 36 issues + softmax + PV 32 issues + TMEM ops)
- Fixed per-outer-iter MMA overhead: alloc/commit/wait/ld ≈ ~200-300 cycles
- Total per outer iter: ~1.5-2.5 µs — **comparable to full current A23 main loop**.
- Main loop total: ~3-5 µs per CTA (REGRESSED).

Why: **compute was not the bottleneck in A23**. A23's per-iter scalar compute
(~400 cycles active) is smaller than the fixed tcgen05 lifecycle overhead
(~200-300 cycles per outer iter × MMA pipeline depth). The 4× padding waste
adds another constant factor. Net result — the MMA path spends more time on
MMA lifecycle and softmax round-trips than A23 spends on scalar compute.

### D3.6 — Also: fundamental regime confirmation

The AI=30 FLOP/B at bf16 tensor-core ridge=714 FLOP/B places the kernel
**23× below** the compute roofline even at MMA precision. Extrapolating
from the A23 measured performance and Amdahl logic: if compute were the
bottleneck, A23 at scalar fp32 (11 FLOP/B ridge) would already be compute-bound
at ~0.2 µs per CTA of scalar compute per outer iter. But A23's per-CTA wall
is ~3-5 µs / 32 iters = ~100-150 ns per outer iter — showing compute is
~30% of active time, and memory+mbar sync is the rest (which tensor cores
do not improve).

Even IF MMA reduced compute to zero, the remaining non-compute fraction
(~70% of wall) would set a floor of ~0.030 ms vs A23's 0.044 ms — best-case
~30% improvement. But the MMA lifecycle cost is exactly the kind of thing
that increases the non-compute fraction. Realistic expected outcome: **regression**.

**V3**: Binding analysis converges on stop-per-criterion. No clean MMA path exists.

## New GT candidate — to be appended to CLAUDE.md §11

The finding below is hardware + regime-specific and reusable for future
MLA-decode kernels with similar shapes. Adding as GT-39.

### GT-39: tcgen05 BF16 MMA regression regime for MLA-decode-shaped kernels

For kernels with:
- Small effective M (number of query heads per contracted MMA, M ∈ {8, 16, 32})
- Split-K structure with many CTAs (~100+) each contributing a small slice
- AI well below the bf16 tensor-core ridge (`< 700 FLOP/byte`)
- Already exploiting TMA + multi-buffer prefetch with memory latency hidden,

moving from per-warp scalar fp32 dot products to `tcgen05.mma.kind::f16` is
expected to **regress** wall time because:

1. `kind::f16` minimum M = 64 per ISA Table 41; M=16 real heads → 4× compute
   padding waste on the MMA accumulator.
2. The fixed per-MMA-group lifecycle (`tcgen05.alloc/commit/mbar_wait/ld/dealloc`,
   ~200-300 cycles per outer iter) is comparable to or exceeds the scalar
   compute it replaces when AI is well below the tensor-core ridge.
3. Token-packing to raise M (Option A in T3-5/6/7 framing) destroys split-K
   SM utilization for small-batch decode workloads.
4. Keeping split-K and zero-padding M within a CTA (Option C) requires
   - a new SBO/LBO sweep for `kind::f16` (prior GT-11 covers `f8f6f4` only),
   - a new lane-distribution derivation for tcgen05.ld at M=64 K=16,
   - full SMEM layout reorganization,
   - ~800-1000 lines of MMA pipeline code,
   with expected net regression.

Conclusion confirmed 2026-04-19 via D3 regime analysis on sparse-attention
kernel at A23 baseline (0.044 ms); prior T3-1/T3-3/T3 reduce-pipeline
attempts (A13/A15/A24/A25) separately confirmed that adding fixed per-CTA
overhead regresses when memory latency is already hidden by 3-buffer TMA prefetch.

For **future kernels with identical regime signature** (small M, small batch
decode, random KV gather, AI < 50 FLOP/B), skip T3-5/6/7 directly — the
finding is deterministic from the ISA + roofline, not a property of this
one kernel.
