**New molecule activated in A24:** "Whole-Split Predicate Hoist" — a pure
PRD × FXP intersection applied at CTA granularity (hoist the entire FXP
tile loop when a warp-uniform predicate says every iteration will produce
-∞). This is the "Path Specialization" corner of PRD × SYM (framework §04)
applied one level up: not fast-path/slow-path per tile but per-CTA.

**V1 (vs files 05, 07, 08):** re-verified. Streaming Gate detection rule
("are the GATE dimension's candidates produced inside the FXP tile loop?")
answer is still NO — candidates are fully materialized in
`sparse_indices[t,*]` before kernel entry. Single-pass Gate spec from 07
does not apply either: Gate is fully pre-evaluated before the kernel.
Neither Gate class applies to this kernel, this remains correct.

## D2 — Structural Analysis (deltas)

**New atom activations in A24:**
- **PRD × FXP at CTA scope**: `smem_any_valid` is a warp-uniform
  broadcast flag predicating a multi-iteration FXP. Strengthens the
  predicate hoist from the A21 pass (which only hoisted per-slot).
- **MEA × MEA**: `smem_any_valid` is 1 byte of new SMEM on top of the
  128-int smem_idx cache. No budget concern (CP_SMEM_BYTES is still
  ~14 KB of 48+ KB limit).

**Access patterns unchanged:**
- Q: per-warp LDG.32 interleaved (A18 layout) — 32 distinct banks
- KV: cp.async.bulk to linear SMEM, per-lane LDG.32 interleaved read
- scratch_o: float2 vectorized STG/LDG, interleaved (A7 layout)
- output: __nv_bfloat162 interleaved STG (A8 layout)
- counter: global atomicAdd + __threadfence (A21 pattern)

**V2:** classifications consistent with D1. No new REL pairs introduced
by the empty-hoist change — the broadcast flag is CTA-local.

### Atom intersections NOT yet exploited on A24

The pool below is what remains. Each is evaluated for expected Δ, effort,
and risk.

**R1. REL × ATO — `cuda::atomic_ref` release semantics (drop `__threadfence`)**
Currently: `__threadfence(); atomicAdd(counter, 1);`. The threadfence
enforces "all prior writes device-visible before the counter increment."
`cuda::atomic_ref<int, thread_scope_device>::fetch_add(1, memory_order_release)`
makes the fence redundant by releasing the memory order on the atomic
itself.
- **Expected Δ**: sub-µs (one fence saved per CTA × S × N CTAs).
- **Effort**: 1-line change.
- **Risk**: memory model subtlety — release orders the atomic before
  subsequent loads, but we need it to order *prior stores* before
  the atomic. `memory_order_release` on a store/RMW does exactly that.
  Correct. But this remains P4 from A21 — still untested.

**R2. MEA × FXP — SW-pipelined reduce s_iter loop**
Currently the per-token reduce loop serially: (LDG.64 scratch_o[s]) →
(o_acc += scale * o_partial). Software-pipeline: prefetch s+1 while
accumulating s. Halves LSU latency exposure.
- **Expected Δ**: 0.5-1 µs per token, only on the last-arriving CTA
  (so N µs total across the grid).
- **Effort**: 20-30 LOC reshuffle of the reduce phase.
- **Risk**: register pressure on the last-arriving CTA (already the
  high-pressure path). Likely modest but the effect on occupancy of
  the split kernel (same __launch_bounds__) is a concern.

**R3. MOR × MEA — bf16 scratch_o** (P7 revisited after A24)
Currently scratch_o is fp32 (N × S × 16 × 512 × 4 = up to ~2 MB per
workload). Halving to bf16 cuts HBM scratch write + read in half.
Split kernel holds fp32 in registers, truncates to bf16 only at store.
Reduce kernel reads bf16, upcasts to fp32 for accumulation.
- **Expected Δ**: hard to predict. HBM scratch is ~20% of total traffic
  (KV is dominant). Halving that portion = ~3-5% wall-clock headroom,
  if the traffic *is* the bottleneck. Could be smaller if the reduce
  phase is latency-bound (small S means N LDGs per warp-head, not
  bandwidth-bound).
- **Effort**: low (cast at STG + cast at LDG; layout already vectorized).
- **Risk**: **numerical**. The reduce-phase accumulation
  `o_acc += scale_b * bf16_to_fp32(o_partial)` trades 16 bits of
  per-split mantissa for 2× memory bandwidth. The harness rel_err
  tolerance is ~1e-2. The Python reference uses fp32 throughout the
  softmax path. This probably still passes, but on edge cases could
  fail on large-S workloads where accumulated rounding error shows up.
  Precedent: A21 D3 pass explicitly skipped this for numerical risk.
  Worth testing once given how cheap the change is.

**R4. AFM × MOR — scratch_o pre-scaled write (skip `* inv_l` per position in reduce)**
Observation: reduce does `o_acc += scale_b * o_partial` then final
output is `o_acc * inv_l`. Since inv_l is warp-uniform (depends only
on token), we can pre-multiply `scale_b *= inv_l` once (32 lanes × 1
shfl) and skip the per-position fmul by inv_l. This fuses two
broadcast scalars into one.
- **Expected Δ**: negligible — 16 per-lane fmul saved × 16 warps per
  CTA × 1 CTA per token = 256 FMUL saved per token. Well under 1 µs.
- **Effort**: 2 lines.
- **Risk**: none.
- **Recommendation**: bundle with any other reduce-phase touch; not
  worth a standalone cycle.

**R5. MOR × FUN — Cluster DSMEM Q-broadcast**
The S CTAs per token each LDG the same Q_nope[t]+Q_pe[t] = 16×(512+64)×2 =
18432 B = 18 KB. At S=16 (N=8, peak case), that's 16×18 KB = 288 KB of
Q-read redundancy across CTAs per token (but actually served by L2 in
practice, not a fresh 288 KB of HBM). With a cluster-of-S and DSMEM,
CTA-0 LDGs Q once from HBM; CTA-[1..S-1] read Q from CTA-0's SMEM via
DSMEM.
- **Expected Δ**: the L2 already deduplicates Q reads across CTAs on
  the same SM, and on B200 Q reads cost is ~18 KB/CTA at 7 TB/s = ~2.6 ns
  per CTA = barely measurable. DSMEM path is ~5-10× cheaper than L2
  Q-read per CTA, but the absolute saving is sub-µs per token.
- **Effort**: **HIGH**. Cluster launch, cluster barrier init, DSMEM
  pointer arithmetic, re-tune SMEM budget (cluster SMEM shared across
  all CTAs in a cluster). Several new GTs likely.
- **Risk**: cluster size capped at 8 on B200 by default (16 with
  `cudaClusterSchedulingPolicyMaximumUtilization`); our S can be 16.
  Would need cluster-size = min(S, 8), wasting some savings.
- **Recommendation**: **skip**. Expected return does not justify a
  Tier 3 cluster restructure.

**R6. MEA × SYM — templated kernel on `kv_per_split`**
Currently `kv_per_split` is a runtime parameter; compiler cannot fully
unroll the outer loop. Templating lets `total_iters` be compile-time,
enabling aggressive unroll + register scheduling. Dispatcher picks from
a handful of specializations (kv_per_split ∈ {32, 64, 128}).
- **Expected Δ**: 1-3% if the outer-loop unroll helps register
  allocation. Could regress due to ICACHE pressure of multiple
  specializations and prologue branch.
- **Effort**: moderate — dispatcher + one extra round of `pragma unroll`
  tuning.
- **Risk**: binary size growth, compile time up, possible regression
  on "wrong" workloads near the boundary.

**R7. PRD × ATO — skip reduce entirely when l_global == 0** (already
partially exploited)
The loop `if (scale_b == 0.0f) continue;` already skips empty splits
*within* the reduce. A24's whole-split hoist means the CTA also skips
the *split* phase when its slice is all -1. The remaining gap: if ALL
splits for a token are empty, the last-arriving CTA still does a
full reduce loop (scanning S empty entries). Could add an early
`if (l_global == 0)` to skip the o-accumulate + final write with zero
output + -inf lse.
- **Expected Δ**: negligible on typical workloads (very few tokens
  have all-empty sparse indices). **But** the current reduce already
  handles empty cleanly: m_global = -inf → scale_s = 0 → l_global = 0
  → inv_l = 0 → o_acc2 stays 0 → bf16 write of 0 → -inf lse. So the
  work *is* already skipped, just not early-exited. Nothing to gain.
- **Recommendation**: skip.

**R8. AFM × MEA — store `smem_any_valid` as a flag in the idx load instead of atomicOr**
The A24 implementation uses `atomicOr(&smem_any_valid, 1)` during the
cooperative idx load. An atomicOr in SMEM is about 2-4 cycles, called
by every thread that sees a valid slot (up to 128 of 512 threads). Could
be replaced by OR'ing into a lane-uniform register then ballot across
the block via a single `__syncthreads_or()` (which exists as a block
vote intrinsic) or by exploiting that `smem_idx` is fully written before
the flag is read — i.e., scan SMEM for any-positive after the barrier
using `__any_sync` + reduction.
- **Expected Δ**: sub-µs. Most CTAs with valid slots hit the atomic
  early and short-circuit thanks to SMEM coalescing, but 128 atomicOrs
  into 1 byte is a known pattern for warp-level contention.
- **Effort**: minor (5 lines).
- **Risk**: none.

**R9. MEA × MOR — reduce SMEM to fit 2 CTAs/SM**
CP_SMEM_BYTES ~14 KB + smem_idx (0.5 KB) + mbar (24 B) + bool (1 B) ≈
14.5 KB per CTA. B200 SMEM is 227 KB per SM. Current occupancy is
**1 CTA/SM** due to `__launch_bounds__(THREADS_PER_BLOCK, 1)`. Dropping
to 2 CTAs/SM would require halving SMEM AND removing the `, 1` from
launch bounds AND keeping register pressure under control. Potential
win: hide TMA latency across CTAs on the same SM.
- **Expected Δ**: uncertain. Session log notes "grid size (128 CTAs for
  N=8) means spreading across 64 SMs instead of 132 actually HURTS."
  Saturating 132 SMs with 1 CTA each seems to be the right regime.
  But for N ≤ 6 (S = 132/N > 22, S capped at 64 → S*N ≤ 64N ≤ 384)
  we already use > 132 CTAs. Splitting CTAs across SMs may help tail
  latency but risks launch-overhead regression as seen in A11.
- **Effort**: high — need to shrink SMEM to ~7 KB (drop buffers to 2,
  or interleave KC + KP in a single buffer).
- **Risk**: A11 (S=132) already regressed; this is a cousin change.
- **Recommendation**: do not revisit until reduce overhead is dealt with.

**R10. PRD × AFM — skip the cooperative idx-load for CTAs with total_iters == 0**
The launcher invariant guarantees `kv_per_split ≤ SMEM_IDX_CAP`, but the
cooperative idx-load still runs SMEM_IDX_CAP iterations for every CTA.
At S=64, last split gets kv_per_split=32 → 32 valid reads + 96 sentinel
writes. The 96 extra writes are cheap (SMEM-resident, no L2 trip), but
the loop bound could be `min(SMEM_IDX_CAP, actual)` + a smaller
sentinel-fill.
- **Expected Δ**: sub-µs; the SMEM writes are cheap.
- **Risk**: none.
- **Recommendation**: bundle with next reduce-phase touch; not standalone.

## D4 — Architecture Specification (A24 FSM, incremental view)

FSM is substantively the A21 FSM with one pre-loop fork added:

```
[CTA entry]
  ├── tid==0 : init mbar[0..2]
  ├── cooperative : load smem_idx[0..127] from idx_t
  │              + atomicOr(smem_any_valid) along the way      ← A24
  └── __syncthreads   (smem_idx + smem_any_valid visible)

[Per-warp Q load — per-lane register Q (A18 interleaved)]

[A24 fork]
  if (total_iters > 0 && smem_any_valid):                      ← A24 hoist
    [Inner loop, 3-buffer cp.async.bulk, total_iters iters]    (A14+A23)
      ├── tid==0 : prefetch it+2 batch (A23 evict_first policy)
      ├── all   : mbarrier_wait_parity(consume_buf)
      ├── per-lane: SMEM → registers (A18 interleaved)
      ├── compute K=4 logits + warp_reduce_sum
      ├── batched online-softmax rescale
      └── update o_acc
  else:
    # running state already correct: row_max=-inf, row_sum=0, o_acc=0

[Scratch write — A7 vectorized float2 interleaved STG]

[A21 fused-reduce fork]
  ├── __threadfence
  ├── tid==0 : atomicAdd(counter, 1) → smem_i_am_last
  ├── __syncthreads
  └── if !smem_i_am_last: return

[Reduce phase (last-arrival only)]
  ├── per-warp: read scratch_m[t,*,h], shfl_xor max
  ├── per-warp: read scratch_l[t,*,h], compute scale_s + l_global
  ├── for s_iter in [0, S): LDG scratch_o, o_acc2 += scale_b * o_partial
  ├── bf16 output write (A8 interleaved)
  ├── lse write
  └── tid==0: counter[t] = 0                                    (A22 persistent)
```

### Stopping criterion

The session log's final conclusion (0.044 ms within ~10-20% of the
theoretical HBM-gather floor for 18 MB of random-scatter KV at 7 TB/s)
is reconfirmed by this pass. **The kernel is at a latency floor, not a
bandwidth floor.** The candidates above are last-mile; the expected
aggregate win of committing all of N1-N5 is in the 3-6% range — another
1-2 µs off the mean.

After N1-N5, a second re-derivation should re-compute the AI and
identify whether the dominant cost has shifted from HBM latency to
something else. Only then reopen Tier 3.

## Why this pass produced fewer candidates than A21's

Three reasons:

1. **A21's top three candidates (P1, P2, P3) are shipped.** The A21
   pass had 7 candidates; 3 committed, 1 regressed (P6 via A25), 3
   remain (P4, P5, P7). This pass adds 3 new ones (R4, R8, R10) all
   small and 1 re-evaluation (R3 bf16 scratch).
2. **Tier 3 was exhausted.** The MMA derivation (`d1_d4_v1_v4_t3_5_6_7.md`)
   + the T3-568 empirical run closed the tensor-core avenue for this
   regime (GT-39).
3. **The kernel is latency-bound, not compute- or bandwidth-bound.**
   Most "new" intersections from matrix 04 (e.g., Compression Staging,
   Tensor Cores) only help in regimes where compute or HBM throughput
   is the binding constraint. Here, per-CTA fixed overhead (mbar waits,
   TMA setup, atomic+fence, last-arrival sync) dominates, and only
   optimizations that shave that overhead (N1, N4, N5) have a positive
   expected value.

## What would materially change the ceiling

Not in the candidate pool above because they require spec changes or
major restructures, but worth naming so a future session doesn't
re-derive them:

- **Sort sparse_indices by KV-page locality before the kernel** — turns
  random gather into range gather. Orthogonal kernel, separate op; would
  ~halve HBM traffic on hot pages.
- **Persistent kernel** across multiple consecutive attention calls
  (same KV cache, different queries) — amortizes TMA setup and keeps
  L2 warm. Requires harness support.
- **FP8 KV cache** — violates the definition JSON (CLAUDE.md §1
  mandates bf16). Spec-locked.

These are ceiling-movers, not candidate-list entries.
