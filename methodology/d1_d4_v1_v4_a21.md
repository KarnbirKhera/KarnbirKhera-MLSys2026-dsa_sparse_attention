# D1-D4 / V1-V4 Derivation Pass on A21 Kernel
# `dsa_sparse_attention_h16_ckv512_kpe64_topk2048_ps64`

Second framework walk, post-A21 (fused split+reduce via last-arrival atomic).
What changed vs the A19 pass:
- Now ONE kernel, not two
- New atom in play: SRG × ATO (Atomic Reduction) for the cross-CTA last-arrival
- REL × FXP × FUN (Pipeline Lift): the two fixed points (split tile loop + reduce s_iter loop) now compose in one kernel
- Launcher overhead newly visible: 3× torch::empty + 1× torch::zeros per call

## D2 — Structural Analysis (delta)

**New nested fixed point in the reduce phase**: `for (s_iter = 0; s_iter < splits_per_token; ++s_iter)` reads scratch_o partials and accumulates into final o_acc. This loop has its own:
- AFM (scratch_o read addressing)
- SRG (sum over splits, weighted by scale_s)
- FXP (bounded by S = 16-64)
- REL (cross-warp shfl_sync to broadcast scale_s)

**New atom activations:**
- ATO (cross-CTA atomicAdd on completion_counters)
- REL (between atomic + threadfence — write→read across CTAs)
- MEA (counter buffer = N×4 bytes; new measure)

**V2**: All classifications consistent.

## D3 — Hardware Binding (post-A21 state)

**Atom intersections — exploited in A21 (newly added since A19 pass):**

| Intersection            | Optimization name              | Where in A21                                                  |
| ---                     | ---                            | ---                                                            |
| SRG × ATO               | Atomic Reduction (Global)      | last-arrival counter via atomicAdd(&counter[t], 1)            |
| REL × FXP               | Pipeline Lift                  | fused 2-kernel → 1-kernel; Pass 2 happens in last-arrival CTA |
| REL × ATO               | Acquire-Release Sync           | __threadfence() before atomicAdd                              |

**Atom intersections — NOT exploited in A21 (NEW candidates since A19 pass):**

1. **MEA × FXP × ATO (Persistent Counter Buffer)**
   - Currently `torch::zeros({N}, opts_i32)` per launch: ~1-2 µs allocation + cudaMemsetAsync.
   - Fix: static device buffer cached across launches (per CLAUDE.md GT-16 — counters with kernel-side reset are inter-kernel persistent and qualify for raw cudaMalloc).
   - Kernel-side reset: last-arrival CTA writes `completion_counters[t] = 0` before exit.
   - Saves ~1-2 µs per call (allocation + zero).

2. **MEA × MOR (Scratch Tensor Consolidation)**
   - 3 separate `torch::empty` calls (scratch_m, scratch_l, scratch_o) — each has allocation overhead.
   - Combine into 1 buffer with manual offset slicing. Saves 2 allocation calls = ~0.5-1 µs per call.

3. **REL × ATO (cuda::atomic_ref with explicit memory_order_release) — eliminate threadfence**
   - Currently: `__threadfence(); atomicAdd(&counter, 1)`.
   - With `cuda::atomic_ref<int, cuda::thread_scope_device>::fetch_add(1, memory_order_release)`, the release semantics make the fence redundant.
   - Saves ~10-20 cycles per CTA (one fence). Per-CTA marginal; cumulative across S × N CTAs is small.

4. **AFM × FXP (L2 Cache Hint on cp.async.bulk for KV)**
   - cp.async.bulk supports `.L2::cache_hint` — could mark KV reads as `evict_first` (no-allocate would be cleanest but evict_first is the standard).
   - KV gather is random scatter; rows are read once per CTA per kernel call, never re-used.
   - Frees L2 capacity for scratch_o reads in the reduce phase.
   - Estimated: 1-3% on bandwidth-limited workloads.

5. **REL × FXP (Software-pipelined reduce s_iter loop)**
   - Currently the reduce reads scratch_o[s_iter] then accumulates. 8 LDG.64 per warp per s_iter × S=16 = 128 LDG per warp. With ~200-cycle L2 latency and ~16-deep LSU queue, expected ~1600 cycles per warp.
   - Software pipelining: prefetch s_iter+1's data while accumulating s_iter. Halves latency exposure.
   - Estimated: 0.5-1 µs per token, only on the LAST CTA per token. Modest.

6. **MEA × FUN (4-buffer prefetch)**
   - A16 went 2→3 buffers and won. Try 3→4? SMEM cost: extra 4.6 KB per buffer = total ~18 KB KV. Fits.
   - TMA latency vs compute ratio: with 3-buffer, hide ~400 cycles of compute. If TMA latency > 600 cycles, more buffers help.
   - Risk: register pressure from buf-state tracking; mbarrier count grows.

7. **PRD × ATO (skip reduce when l_global == 0 already in scratch_l)**
   - All splits empty → all l_partial = 0. Last CTA could skip the reduce entirely (write 0 + -inf directly).
   - But the CTA still needs to read scratch_l to detect this. Then skip the o_acc combination.
   - Already implicit: reduce loop's `if (scale_b == 0.0f) continue;` skips empty splits. Compute savings minimal.
   - Tiny gain; skip.

8. **MOR × MEA (Templated kv_per_split for compile-time outer loop unroll)**
   - kv_per_split is a runtime parameter; compiler can't fully unroll the outer loop.
   - Templating on kv_per_split (specialize kernel per workload) lets the compiler unroll.
   - Trade-off: launch dispatcher with switch statement; bigger binary.
   - Estimated: 1-2% if unroll helps register allocation; risk of compile time blowup.

**V3**: K/N ratio still ≈ 4 (k=2048, output ≈ 512). Online algorithm correct. Binding table complete. New ATO intersection now in use; persistent counter is the obvious next intersection to exploit.

## D4 — Architecture Specification (post-A21 state)

**FSM (fused kernel):**
```
[CTA entry]
  ├── tid==0 : init mbar[0..2]                                       (REL setup)
  ├── all   : cooperative load smem_idx[0..127] from idx_t           (MEA × MOR)
  ├── tid==0 : issue cp.async.bulk batch 0, batch 1                  (FXP prologue)
  └── __syncthreads                                                   (Q load needs nothing; mbar visible)
[Per-warp Q load — per-lane register Q (interleaved layout)]
[Inner loop, total_iters iters]
  ├── tid==0 : if has_issue → cp.async.bulk batch (it+2) → buf       (FXP prefetch)
  ├── all   : mbarrier_wait_parity(consume_buf, phase[buf])          (REL: TMA done)
  ├── per-lane: SMEM → registers (interleaved kc_local + kp_local)   (AFM × SYM)
  ├── compute K independent logits (warp_reduce_sum)                 (SRG × FREE TREE)
  ├── batched online-softmax rescale                                 (FXP × SRG × INV)
  └── update o_acc                                                    (SRG × INV)
[Scratch write — interleaved STG.64]                                  (AFM × SYM)
[ATOMIC LAST-ARRIVAL]                                                  ← new in A21
  ├── __threadfence                                                   (REL ordering)
  ├── tid==0: atomicAdd(&counter[t], 1) → smem_i_am_last              (SRG × ATO)
  └── __syncthreads                                                    (broadcast i_am_last)
[If !i_am_last: return]
[Reduce phase — only the last-arriving CTA per token reaches here]
  ├── per-warp: read scratch_m[t,*,h], compute m_global               (SRG × MON-like via shfl_xor)
  ├── per-warp: compute scale_s, l_global                             (SRG × INV)
  ├── for s_iter in [0, S):                                           (FXP — INNER fixed point)
  │   ├── load scratch_o[t, s_iter, h, *] (interleaved LDG.64)
  │   └── o_acc += scale_b * o_partial
  ├── final bf16 output write (interleaved STG.32)
  └── lse write
```

**Lifetime tables (added since A19):**

| Variable              | Birth                | Death                 | Storage          | Size                 |
| ---                   | ---                  | ---                   | ---              | ---                  |
| smem_i_am_last        | post-atomic broadcast | end of reduce phase  | shared mem       | 1 byte               |
| m_global              | reduce phase         | end of CTA            | registers        | 1 fp32/lane (warp-uniform) |
| l_global              | reduce phase         | end of CTA            | registers        | 1 fp32/lane (warp-uniform) |
| inv_l                 | reduce phase         | end of CTA            | registers        | 1 fp32/lane          |
| o_acc2                | reduce phase         | end of CTA            | registers        | 16 fp32/lane         |
| scale_s               | reduce phase         | end of CTA            | registers        | 1 fp32/lane (lane=split holds value) |

**Per-launch host allocation lifecycle:**
- scratch_m, scratch_l: allocated, used, freed via torch ref counting
- scratch_o: same
- counters: torch::zeros (allocate + zero)

This launcher work is now a measurable fraction of small-N wall time (~3-5 µs per call).

**V4**: FSM matches Online + Atomic-Reduce + Pipeline-Lift molecules. Address composition correct (scratch_o offset uses int64_t multiply for safety). Threadfence justifies cross-CTA visibility. Reduce phase has its own sequential FXP (s_iter loop) — opportunity for software pipelining noted above.

## Optimization Candidates Ranked (NEW since A21)

| # | Candidate                                                                  | Expected Δ mean | Effort   | Risk                                |
|---|----------------------------------------------------------------------------|-----------------|----------|-------------------------------------|
|**P1**| **Persistent counter buffer (cudaMalloc once + kernel-side reset)**     | -1 to -2 µs     | Low      | Static state lifetime; thread-safe? |
|**P2**| **Scratch tensor consolidation (1 alloc, 3 views)**                     | -0.5 to -1 µs   | Low      | None; pure launcher cleanup         |
|**P3**| **L2 cache hint `evict_first` on KV cp.async.bulk**                     | -1 to -3%       | Low      | Wrong hint → cache thrashing        |
|**P4**| **cuda::atomic_ref + memory_order_release (drop __threadfence)**         | sub-µs          | Low      | Memory model nuance                 |
|**P5**| **Software-pipelined reduce s_iter loop (prefetch s+1 during s)**       | -0.5 to -1 µs   | Moderate | Register pressure on last CTA       |
|**P6**| **4-buffer prefetch (extend A16's 3-buffer to 4)**                      | uncertain       | Low      | Mbarrier complexity, SMEM ↑         |
|**P7**| **Template kernel on kv_per_split (compile-time unroll)**               | 1-2% if helps   | High     | Binary size, dispatcher complexity  |

**Recommendation order:**
1. **P1 + P2** as a bundled launcher cleanup (one attempt covers both).
2. **P3** L2 cache hint — single-line PTX modifier change.
3. **P4** atomic_ref — minor but principled.
4. **P5** software pipelining if the reduce phase is exposed.
5. **P6** 4-buffer if (5) doesn't move the needle.
6. **P7** templated unroll only if all above plateau.

After P1-P5, the kernel will be at HBM-gather-latency ceiling. P6/P7 are last-mile.

---

## Why these candidates and not others?

**Skipped:**
- **MOR Compression Staging (bf16 scratch)**: scratch is fp32 for precision; reducing to bf16 causes ~1e-2 numerical drift, near the harness tolerance. High risk of failing some workloads.
- **PRD × FXP Predicate Hoisting (already done in A19)**: maxed out.
- **SYM Pipeline period symmetry**: 3-buffer is period-3; period-2 was tried in A14 and beaten by A16; period-4 is P6 candidate.
- **MOR × FUN Kernel Fusion (Cluster + DSMEM Q-share, T3-4)**: rejected with reasoning in A21 round; ROI too small.

**Conclusion**: the framework re-walk surfaces 7 new candidates, of which 5 are low-effort. The highest-ROI next attempt is the **launcher cleanup (P1 + P2)** since per-call allocation overhead is now a meaningful fraction of the 47 µs mean wall time.
